Critique — Sécurité bloquante 7 points ▼
01
Révoquer et régénérer tous les secrets exposés dans le repo
Le fichier .env.local contient les vraies clés API committées : ANTHROPIC_API_KEY, SUPABASE_SERVICE_ROLE_KEY (bypass RLS total), CLOUDMERSIVE_API_KEY. Révoquer immédiatement dans les dashboards Supabase et Anthropic.
Sécurité Secrets
.env.local · Supabase Dashboard · Anthropic Console
02
Réécrire les policies RLS de la table clients
Actuellement using (true) pour SELECT, INSERT, UPDATE, DELETE — n'importe quel user connecté voit et modifie toutes les données. Remplacer par des conditions basées sur le rôle ou created_by.
Sécurité RLS
lib/supabase-rls-policies.sql
03
Réécrire les policies RLS de la table soumissions
Même problème : using (true) sur toutes les opérations. Un commercial peut lire toutes les soumissions des autres utilisateurs, les modifier, les supprimer.
Sécurité RLS
lib/supabase-rls-policies.sql
04
Réécrire les policies RLS de la table lignes_budget
using (true) sur toutes les opérations. N'importe qui peut modifier les lignes budget de n'importe quelle soumission d'un autre utilisateur.
Sécurité RLS
lib/supabase-rls-policies.sql
05
Ajouter vérification d'authentification sur GET /api/prospects
Le handler GET n'a aucun check d'auth. N'importe qui sans token peut lire tous les prospects. L'auth n'existe qu'au POST (ligne ~46). Déplacer getUser() en tête du GET.
Sécurité Auth
app/api/prospects/route.ts · ligne 25
06
Ajouter vérification d'authentification sur GET /api/clients
