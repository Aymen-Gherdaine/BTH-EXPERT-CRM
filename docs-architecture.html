Vue d'ensemble du projet

En une phrase

BTH Hub est le CRM interne de BTH Expert, un cabinet algérien de conseil en environnement. Il couvre le cycle complet de vente : de la prospection terrain jusqu'à la remise d'un document de soumission professionnel en DOCX/PDF, avec rédaction automatique par IA (Claude Anthropic).

  • Formulaire 4 étapes (wizard)
  • Génération IA du contenu FR
  • Aperçu éditable en temps réel
  • Export DOCX + PDF professionnel
  • Suivi statut (Brouillon → Acceptée)
  • Numéro offre auto (T110520.2026)

👥 Clients

  • Annuaire entreprises/contacts
  • Recherche par nom ou entreprise
  • Lié aux soumissions (cascade)
  • Export XLSX

🔍 Prospection

  • Alertes relances overdue

💸 Dépenses

💡 Server vs Client — règle générale

Server Components

Pages (page.tsx), layouts, composants qui affichent uniquement des données. Accès direct à Supabase server-side, pas de JS envoyé au client.

Client Components "use client"

Tout ce qui a des useState, useEffect, des event handlers (onClick, onChange), ou des hooks Next.js (usePathname). Tout le wizard soumission, formulaires, navigation.

🛡️ Comment le RBAC est appliqué

1. Middleware (Edge)

Intercepte toutes les requêtes. Vérifie la session Supabase + le rôle depuis profiles. Redirige si non autorisé avant même de rendre la page.

2. Layout (Server)

Le layout protégé charge le profil et filtre la Sidebar selon le rôle. Double protection pour les routes sensibles.

3. API Routes (Server)

Chaque route API vérifie createServerClient + cookies() + getUser() explicite. Routes admin vérifient le rôle. RLS Supabase comme dernière ligne de défense. Session 9 : auth ajoutée sur toutes les routes manquantes.

Guide de présentation — Entrevue

Présentation en 30 secondes

"J'ai développé BTH Hub, un CRM interne pour un cabinet de conseil en environnement. L'application couvre tout le cycle de vente : prospection terrain, création de soumissions techniques avec rédaction assistée par IA, et génération de documents professionnels Word/PDF. C'est une application Next.js full-stack avec Supabase comme backend, et Claude d'Anthropic pour la génération de contenu."

Question : "Qu'est-ce que tu es le plus fier dans ce projet ?"

Le pipeline de génération de documents. L'utilisateur remplit un formulaire, Claude génère un contenu professionnel en français avec le style BTH Expert, l'utilisateur édite en temps réel, et on exporte un Word ou PDF en un clic via un vrai template DOCX avec docxtemplater. Zéro jsPDF, le rendu est identique à un document Word natif.

Question : "Comment as-tu géré la sécurité ?"

Trois couches : le middleware Edge intercepte toutes les routes et vérifie le rôle avant le rendu. Les API routes créent un client Supabase côté serveur avec les cookies de session — jamais le client anonyme. Et Supabase RLS comme dernière défense au niveau base de données.

Question : "Server Components vs Client Components ?"

Les pages sont Server Components par défaut : elles lisent Supabase directement côté serveur, sans hydratation client. Je passe en "use client" uniquement quand c'est nécessaire : formulaires avec état, animations Framer Motion, hooks usePathname. Le wizard soumission (4 étapes) est entièrement client car il gère un état complexe inter-steps.

Question : "Comment fonctionne l'IA ?"

L'utilisateur remplit les étapes 1 (client) et 2 (projet). À l'étape 3, quand il valide le budget, on appelle POST /api/generate qui envoie les données à Claude Sonnet avec un prompt structuré incluant les guidelines stylistiques BTH Expert. Claude retourne un JSON avec 10+ champs textuels en français. L'utilisateur peut tout éditer avant export.

Session 9 — Architecture & Sécurité (16 mai 2026)

🏗️ Migration Architecture — Shell SSR + Client Fetch

Un CRM privé n'a pas besoin de SSR pour le contenu — il n'y a ni SEO ni utilisateurs anonymes. L'architecture full SSR créait une latence perceptible à chaque changement de section car Next.js re-rendait les pages côté serveur à chaque navigation. Elle provoquait aussi des erreurs d'hydratation (useBp() rendait "desktop" côté serveur, "mobile" côté client).

❌ Avant — Full SSR
Middleware — Vérifie auth (Edge)
↓ round-trip
Layout SSR — Fetch profil + données layout côté serveur
↓ round-trip
Page SSR — Fetch données (soumissions, dashboard…) — re-exécuté à chaque navigation
↓ round-trip
Client render — Hydratation + risque de mismatch SSR/Client
⚠ Chaque navigation = nouveau round-trip serveur complet
⚠ Hydration mismatch sur useBp() (server="desktop" vs client="mobile")
⚠ Latence perceptible entre sections
✅ Après — Shell SSR + Client Fetch
Middleware — Vérifie auth (Edge) — inchangé, getUser()
↓
Layout Shell SSR — Rendu instantané (sidebar, navbar) — zéro fetch de données
↓
Page — thin wrapper → return <DashboardClient /> (pas d'async)
↓ useEffect au mount
Promise.all([fetch("/api/me"), fetch("/api/dashboard"), …]) — chargement parallèle en arrière-plan
✓ Navigation instantanée (chrome déjà rendu, pas de re-fetch)
✓ Zéro hydration mismatch (Client Component uniquement)
✓ Rôle + données chargés en parallèle via Promise.all

🐛 Bug corrigé — Hydration mismatch useBp()

❌ Avant 
const [bp, set] = useState(
  typeof window === "undefined"
    ? "desktop"   // SSR → "desktop"
    : getBp()     // Client → peut être "mobile"
);

Le serveur rend "desktop", le client hydrate en "mobile" → React Hydration Error

✅ Après 
const [bp, set] = useState("mobile");
useEffect(() => {
  const h = () => set(
    window.innerWidth >= 1024 ? "desktop"
    : window.innerWidth >= 640 ? "tablet"
    : "mobile"
  );
  h(); // calcul immédiat post-mount
  window.addEventListener("resize", h);
  return () => window.removeEventListener("resize", h);
}, []);

Init stable "mobile" côté serveur et client → mise à jour uniquement après mount

🛡️ Modèle de sécurité — 3 couches de défense

🔒
Couche 1 — Middleware Edge

auth.getUser()
Validation JWT côté serveur Supabase Auth.
Redirige → /login si token invalide ou expiré.

Corrigé session 9 : getSession() faisait confiance au JWT local sans validation réseau — un token forgé passait.
🔑
Couche 2 — API Routes

getUser() explicite sur chaque route.
Retourne 401 si non authentifié.
Indépendant du middleware — défense en profondeur.

Session 9 : auth ajoutée sur 10+ routes qui dépendaient uniquement du RLS. Export routes étaient 100% non protégées.
🗄️
Couche 3 — RLS Supabase

Row Level Security au niveau base de données.
Même si couches 1 et 2 échouent, les données restent inaccessibles.

WITH CHECK obligatoire en plus de USING pour les inserts. Commercial voit uniquement ses propres entrées.

👤 Nouvelle route — GET /api/me

Permet aux Client Components de récupérer leur rôle et nom sans avoir à recevoir ces données en props depuis un Server Component. Utilisée par DashboardClient et SoumissionsClient en parallèle avec les autres fetches dans Promise.all.

  • → getUser() — validation auth
  • → profiles.select("role, full_name")
  • → Retourne { role, full_name }
  • → Fallback : role: "admin" si profil absent
Utilisation dans DashboardClient :
Promise.all([
  fetch("/api/me").then(r => r.json()),
  fetch("/api/dashboard").then(r => r.json()),
  fetch("/api/soumissions").then(r => r.json()),
  fetch("/api/prospects?statut=actif").then(r => r.json()),
]).then(([me, stats, soum, prospects]) => {
  setRole(me.role ?? "admin");
  setUserName(me.full_name || "Utilisateur");
  // ...
}).catch(() => setLoading(false));
